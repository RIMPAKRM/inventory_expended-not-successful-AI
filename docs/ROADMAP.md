# ROADMAP — Extended Inventory Core

## Область roadmap

Этот roadmap покрывает только ядро мода инвентаря/экипировки:
- серверно-авторитетное хранение,
- динамические слоты от экипировки,
- производительная синхронизация,
- API для внешних модов,
- игровой GUI полной замены ванильного UX инвентаря,
- админский GUI (онлайн/офлайн).

Не входит в этот roadmap:
- генерация лута,
- редкие материалы,
- интеграция лут-логики,
- «совместимость со storage-модами» как центральная цель.

---

## Формат этапов

Для каждого этапа фиксируем:
- цель,
- deliverables,
- acceptance criteria,
- риски,
- зависимости.

Ориентир по срокам: **8-10 недель**, спринты по 1-2 недели.

---

## Этап 1 (Недели 1-2): Foundation хранения и жизненного цикла выполнено

### Цель
Поднять стабильное серверное хранение и lifecycle игрока без потери данных.

### Deliverables
- `Player Capability` для runtime-работы с инвентарём.
- `SavedData` по UUID для persistent/offline-хранения.
- Базовая сериализация/десериализация NBT (`schemaVersion`).
- Lifecycle-обработчики: login/logout/clone/respawn/dimension change.
- `revision` и `dirty`-модель состояния.

### Acceptance Criteria
- После релога инвентарь полностью восстанавливается.
- После respawn/clone данные не дублируются и не пропадают.
- После смены измерения состояние корректно синхронизируется.
- Офлайн-данные игрока доступны на сервере без загрузки сущности в мир.

### Риски
- Некорректный `clone` может вызвать потерю или дублирование предметов.
- Ошибки сериализации NBT могут ломать обратную совместимость.

### Зависимости
- Базовый каркас мода Forge 1.20.1.

---

## Этап 2 (Недели 2-4): Универсальная модель слотов и layout выполнено

### Цель
Сделать динамическую слот-систему, независимую от фиксированного количества слотов.

### Deliverables
- Модель `SlotDefinition` и `SlotGroupDefinition`.
- Resolver layout на основе экипировки и конфигурации.
- Поддержка источников слотов: база, экипировка, контейнер.
- Кэш resolved-layout и invalidation по событиям.

### Acceptance Criteria
- Количество и типы слотов меняются от надетой экипировки.
- GUI не ломается при увеличении числа слотов.
- Неактивные группы не накладываются на активные.
- Пересборка layout происходит только по изменяющим событиям.

### Риски
- Сложные layout-конфликты между группами слотов.
- Ошибки invalidation-кэша приводят к устаревшему отображению.

### Зависимости
- Этап 1 (`revision`, capability, persistent storage).

---

## Этап 3 (Недели 4-5): Транзакции и безопасность операций

### Цель
Исключить потерю предметов при любых изменениях экипировки/слотов.

### Deliverables
- `InventoryTransactionService` с `snapshot -> dry-run -> commit/rollback`.
- Политики overflow:
  - `deny_remove_if_overflow`
  - `move_to_available_slots_first`
  - `allow_overflow_readonly`
  - `drop_to_world_last_resort` (опционально)
- Серверная валидация всех операций инвентаря.

### Acceptance Criteria
- Снятие экипировки с занятыми зависимыми слотами не приводит к потере предметов.
- При неуспешной операции состояние полностью откатывается.
- Клиент не может обойти проверку серверной логики.

### Риски
- Частичный коммит при ошибках в цепочке переноса.
- Некорректный rollback для сложных перемещений.

### Зависимости
- Этап 2 (актуальная модель слотов/layout).

---

## Этап 4 (Недели 5-6): Синхронизация без перегруза сервера

### Цель
Сделать устойчивую event-driven синхронизацию без packet flood.

### Deliverables
- `S2CFullInventorySyncPacket` (полные синки по событиям).
- `S2CPartialInventorySyncPacket` (dirty slots/groups).
- Агрегация изменений за тик (batching).
- Правила отправки sync:
  - full sync: login/open menu/clone/dimension change/layout change,
  - partial sync: локальные изменения содержимого.

### Acceptance Criteria
- Нет постоянной отправки полных пакетов по тикам.
- Изменения нескольких слотов отправляются одним агрегированным пакетом.
- Клиент после серии операций получает консистентное состояние.

### Риски
- Пересинхронизация при конкурирующих источниках изменений.
- Потеря частичных апдейтов без корректной `revision`-проверки.

### Зависимости
- Этапы 1-3.

---

## Этап 5 (Недели 6-8): API для внешних модов

### Цель
Открыть безопасный серверный API для модов, которые используют инвентарь.

### Deliverables
- `IExtendedInventoryApi` (read/search/mutate операции).
- Запросы доступа и контроль прав операций (read-only/read-write).
- Транзакционные операции извлечения/вставки/резерва.
- Документация интеграции для внешних модов.

### Acceptance Criteria
- Мод-потребитель может найти и изъять ресурсы только через API.
- Прямой доступ к внутреннему `ItemStackHandler` не требуется и не выдаётся.
- Все изменения через API отражаются в `revision` и sync.

### Риски
- Неограниченный API может дать обход серверных ограничений.
- Некорректные фильтры поиска приведут к ложным операциям.

### Зависимости
- Этап 3 (транзакции), Этап 4 (синхронизация).

---

## Этап 6 (Недели 8-10): Админский GUI и офлайн-операции

### Цель
Дать администраторам инструмент безопасного управления инвентарём игроков.

### Deliverables
- `AdminPlayerInventoryMenu` и `AdminPlayerInventoryScreen`.
- Поиск игрока по имени/UUID.
- Online/offline режим редактирования.
- `AdminInventoryService` с permission checks.
- Audit log админских действий.
- Обработка конфликтов версий для офлайн-редактирования.

### Acceptance Criteria
- Админ может изъять/выдать/переместить предметы игроку онлайн.
- Админ может безопасно редактировать инвентарь офлайн-игрока.
- При конфликте версий операция корректно отклоняется или повторяется.
- Все админские действия логируются.

### Риски
- Race condition между входом игрока и офлайн-правкой.
- Недостаточный контроль прав в админских операциях.

### Зависимости
- Этап 1 (SavedData + revision), Этап 4 (sync), Этап 5 (service/API слой).

---

## Этап 7 (Недели 10-12): Production UI Replacement (игровой инвентарь)

### Цель
Полностью заменить ванильный экран инвентаря на production-ready интерфейс нового инвентаря без ослабления серверной авторитетности и без packet flood.

### Deliverables
- `ExtendedInventoryMenu` и `ExtendedInventoryScreen` как основной пользовательский инвентарь игрока.
- Клиентский роутинг открытия инвентаря (вместо ванильного экрана) с feature flag и kill switch.
- `C2SInventoryActionPacket` + серверный action-pipeline (валидация, транзакция, commit/rollback, ответ клиенту).
- Контракт совместимости клиент/сервер: `schemaVersion` + `layoutVersion` + `revision` для full/partial sync.
- Защита от packet flood:
  - клиент не спамит действиями быстрее серверного budget,
  - сервер агрегирует sync и отклоняет некорректные/устаревшие действия.
- Набор production-тестов:
  - unit для action-validation/transaction boundaries,
  - интеграционные сценарии GUI->server,
  - soak/smoke проверки под серией быстрых операций.

### Acceptance Criteria
- При нажатии открытия инвентаря игрок стабильно видит новый `ExtendedInventoryScreen`, а не ванильный экран.
- Все перемещения/клики проходят только через серверную валидацию; клиентские попытки обхода отклоняются.
- Операции с предметами транзакционны: нет частичных коммитов и потерь предметов.
- Full/partial sync работает консистентно с проверкой `schemaVersion/layoutVersion/revision`.
- При burst-действиях нет packet flood и нет деградации TPS из-за sync.
- Есть контролируемый rollback: переключение feature flag возвращает ванильный UI без повреждения данных.

### Риски
- Конфликты UX-потока открытия инвентаря с ванильными/сторонними хендлерами клавиш.
- Регрессии drag-and-drop/quick-move поведения по сравнению с ожиданиями игрока.
- Рост сетевой нагрузки при неверном throttling клиентских action-событий.

### Зависимости
- Этапы 1-4 (данные, layout, транзакции, стабильный sync).
- Этап 5 (service/API слой для единой точки мутаций).

---

## Этап 8+ (Недели 12-14): UX Expansion (персонаж/экипировка + встроенный крафт)

### Цель
Улучшить usability нового инвентаря: добавить центральный блок персонажа с экипировкой вокруг силуэта и встроенную крафт-панель справа без ослабления server-authoritative модели.

### Deliverables
- Новый layout `ExtendedInventoryScreen`:
  - центр: силуэт игрока,
  - слоты экипировки вокруг,
  - правая панель крафта в ванильной стилистике.
- Базовые слоты экипировки:
  - голова,
  - лицо,
  - верх,
  - жилет,
  - рюкзак,
  - перчатки,
  - штаны,
  - ботинки.
- Режимы отображения инвентаря:
  - `creative`: показывается полный ванильный инвентарь (`main 3x9` + `hotbar 1x9`),
  - `survival/adventure`: показывается модовый инвентарь, где из ванильных слотов доступен только `hotbar 1x9`.
- Дополнительные слоты от экипировки (карманы/utility и т.п.) появляются рядом с основным инвентарем в том же окне и выглядят как обычные ванильные слоты.
- Встроенная крафт-панель:
  - категории рецептов слева, сетка рецептов справа,
  - доступные рецепты подсвечены, недоступные затемнены,
  - `tooltip`: название, требования, текущие ресурсы,
  - кнопка `Создать` активна только при наличии всех материалов.
- Поведение dynamic slots при экипировке:
  - layout пересчитывается на сервере,
  - при смене layout отправляется `LAYOUT_CHANGED` full sync,
  - появление/исчезновение слотов только после server sync.

### Acceptance Criteria
- `ExtendedInventoryScreen` стабильно открывается с новым M8+ layout.
- В `creative` отображается полный ванильный инвентарь (`main 3x9` + `hotbar 1x9`).
- В `survival/adventure` из ванильных слотов доступен только hotbar; дополнительные слоты корректно добавляются/убираются экипировкой после server sync.
- Слоты экипировки и дополнительные dynamic слоты корректно обновляются при смене экипировки без потери предметов.
- Правая крафт-панель корректно отражает доступность рецептов и блокирует `Создать` при нехватке ресурсов.
- Все действия остаются server-authoritative и не вызывают packet flood.

### Риски
- Перегрузка экрана при большом числе дополнительных слотов.
- UX-регрессии относительно ванильного флоу.
- Ошибки доступности рецептов при stale client state.
- Рост full sync при частой смене экипировки.

### Зависимости
- Этап 7 (production UI replacement).
- Этапы 3-4 (транзакционная целостность и стабильный sync).

## Сквозные задачи качества (на всех этапах)

### Тестирование
- Unit-тесты на чистую логику:
  - перенос и валидация,
  - rollback,
  - миграции schema/layout,
  - ревизии и конфликтные записи.
- Интеграционные сценарии:
  - login/logout,
  - clone/respawn,
  - dimension change,
  - админ online/offline операции,
  - операции внешнего API.

### Наблюдаемость и эксплуатация
- Логирование без спама (info/warn/error + debug toggle).
- Отдельный audit log админских действий.
- Метрики/счётчики (минимум):
  - число full/partial sync,
  - средний размер sync-пакета,
  - число rollback-операций,
  - число конфликтов версий.

### Миграции и совместимость данных
- Поддержка `schemaVersion` и `layoutVersion`.
- Явные миграции при изменении структуры слотов.
- Безопасный режим для «устаревших» слотов (overflow/read-only).

## Контрольные точки (Milestones)

- Runtime + persistent хранение готовы.
- Lifecycle и NBT стабильно работают.

- Универсальные слоты и resolver работают.

- Snapshot/dry-run/rollback покрывают ключевые операции.

- Event-driven sync работает без packet flood.
- Статус итерации (2026-03-12): **completed**.

## Definition of Done для версии v1

Версия считается готовой, если:
- сервер остаётся источником истины для всех операций,
- динамические слоты от экипировки работают стабильно,
- нет потери предметов при смене layout/экипировки,
- синхронизация не перегружает сервер,
- внешний API доступен и безопасен,
- пользовательский интерфейс инвентаря полностью заменяет ванильный экран в production-режиме,
- администратор может управлять инвентарём игроков онлайн и офлайн,
- миграции данных проходят без разрушения существующих сейвов.
