# Architecture Notes — Extended Inventory Core

## 1. Что фиксируем как актуальную область мода

Этот мод — **основной мод хранения и экипировки игрока** для сервера на **Minecraft Forge 1.20.1**, **Java 17**, **Gradle**.

### Цели
- Сделать **серверно-авторитетный** инвентарь игрока.
- Хранить все данные **на сервере**.
- Построить **универсальную систему слотов**, где экипировка может добавлять новые слоты.
- Поддержать **свою одежду и броню**: куртки, бронежилеты, разгрузки, противогазы и другие предметы экипировки.
- Дать другим модам безопасный способ **читать и изменять инвентарь через API**, если им это нужно для своей логики.
- Сделать **игровой production GUI**, который заменяет ванильный экран инвентаря без изменения низкоуровневых vanilla-структур хранения.
- Сделать **админский GUI** для просмотра и изменения инвентаря игрока, в том числе если игрок офлайн.
- Не перегружать сервер постоянной синхронизацией и лишними пакетами.

### Важные уточнения
- Инвентарь **всегда доступен**, потому что фактически является заменой стандартному UX хранения.
- Количество слотов **не фиксировано**. Оно зависит от конфигурации и экипировки.
- Отображение слотов должно быть **универсальным**, чтобы новые категории и группы не накладывались друг на друга.
- Другие моды **не являются центром совместимости**. Этот мод — основной. Но внешние системы могут обращаться к его инвентарю.

---

## 2. Что вычёркиваем из прежнего анализа

### Что остаётся как полезная оговорка
- Некоторые моды могут **обращаться к инвентарю** игрока:
  - мод на оружие может искать модули,
  - система радиации может проверять наличие и состояние противогаза,
  - медицинский мод может работать с подсумками/контейнерами.

## 3. Основная архитектурная идея

### Ключевой принцип
**Источник истины — сервер.**
Клиент только отображает состояние и отправляет намерения пользователя.

### Не делать
Не стоит пытаться буквально переписать всю ванильную `Inventory` игрока как низкоуровневую замену всех внутренних структур Minecraft.

### Делать лучше
Сделать **единый инвентарный слой мода**, который:
- включает ванильные зоны в едином GUI,
- добавляет собственные слоты экипировки и хранения,
- хранит расширенные данные на сервере,
- предоставляет серверный API для чтения/изменения содержимого,
- перехватывает открытие инвентаря и показывает `ExtendedInventoryScreen` как основной пользовательский интерфейс.

Это даёт полную замену UX ванильного инвентаря в production, но без риска ломать базовые внутренние механики Minecraft и Forge.

## 4. Рекомендуемая модель данных

### 4.1. Два уровня хранения

#### A. Runtime-уровень игрока
`Player Capability` + `ItemStackHandler`

Нужен для:
- работы с открытым меню,
- быстрого доступа к инвентарю онлайн-игрока,
- серверной логики предметов и экипировки.

#### B. Постоянное хранилище сервера
`SavedData` по UUID игрока

Нужно для:
- офлайн-доступа из админского GUI,
- безопасного редактирования инвентаря игрока без его входа на сервер,
- централизованной сериализации/миграции,
- более надёжной модели сохранения.

### Почему это важно
Если хранить всё только в capability игрока, офлайн-администрирование становится неудобным и рискованным.

Лучше так:
- при входе игрока данные загружаются из `SavedData` в capability,
- при изменении ставится `dirty`-флаг,
- при выходе/сохранении capability сбрасывается обратно в `SavedData`.

Это делает офлайн-редактирование нормальной поддерживаемой функцией, а не хаком через прямую правку player `.dat`.

## 5. Универсальная система слотов

Нельзя проектировать слоты как просто массив фиксированной длины с жёстко пришитым GUI.

### Нужна модель `SlotDefinition`
Для каждого слота или группы слотов должны быть описаны:
- `slotId`
- `slotType`
- `displayGroup`
- `order`
- `acceptRules`
- `source` — базовый слот, слот от экипировки, слот от контейнера, временный слот
- `visible`
- `enabled`
- `uiAnchor`
- `uiPriority`

### Пример категорий
- `BODY_OUTERWEAR` — куртка
- `BODY_ARMOR` — бронежилет
- `CHEST_RIG` — разгрузка
- `FACE_MASK` — противогаз
- `BACK` — рюкзак
- `BELT` — пояс
- `UTILITY` — подсумки / универсальные контейнеры
- `PATCH` / `RADIO` / `MEDICAL` — специальные слоты при необходимости

### Как должна работать экипировка
Экипируемый предмет должен уметь предоставлять:
- собственные слоты,
- ограничения на типы предметов,
- визуальную группу в GUI,
- приоритет расположения,
- возможные бонусы/правила.

Пример:
- разгрузка добавляет 6 слотов под магазины и 2 универсальных;
- бронежилет добавляет защиту, но сам по себе не обязан давать место;
- куртка может давать 2 кармана;
- рюкзак может открывать отдельный блок слотов.

## 6. Как сделать GUI универсальным и не допустить наложения

Жёстко захардкоженный GUI быстро сломается.

### Нужен layout engine уровня меню/экрана
Минимум нужна схема:
- `layout sections`
- `slot groups`
- `anchors`
- `rows/columns`
- `scrollable blocks` или страницы

### Практичное решение
Разделить экран на панели:
- панель экипировки,
- панель карманов/контейнеров,
- панель ванильного инвентаря,
- опционально панель админских действий.

Если слотов становится слишком много:
- включать прокрутку,
- группировать по секциям,
- скрывать неактивные группы,
- показывать заголовок группы и источник слотов.

### Важный вывод
Отображение должно строиться **на основе описания слотов**, а не на основе жёсткой карты координат под один конкретный набор экипировки.

### Production-требование к UX
Новый экран должен стать **дефолтным экраном инвентаря игрока** (вместо ванильного) после rollout, а не отдельным тестовым окном.

## 6.1. Production-замена ванильного UI: контракт и эксплуатация

### Компоненты UI-замены
- `ExtendedInventoryMenu` — серверно-авторитетный контейнер операций.
- `ExtendedInventoryScreen` — клиентский экран отображения.
- `InventoryScreenRouter` — переключение открытия инвентаря на новый экран.
- `InventoryUiFeatureFlags` — флаги rollout/kill-switch для безопасного включения и отката.

### Контракт клиентских действий
Клиент отправляет только намерения (`intent`), а не состояние.
Минимальный контракт `C2SInventoryActionPacket`:
- `actionId` (уникальный id действия),
- `playerRevision` (какую ревизию видит клиент),
- `layoutVersion`,
- тип действия (`click/quick_move/swap/drop/split`),
- аргументы слотов.

Серверный pipeline:
1. Проверка `schemaVersion/layoutVersion/revision`.
2. Проверка валидности действия и прав доступа.
3. Транзакция `snapshot -> dry-run -> commit/rollback`.
4. Обновление `revision`.
5. Отправка `partial/full sync` по правилам batching.

### Anti-flood и сетевой budget
- Клиент не отправляет повторные action-пакеты без изменения input-состояния.
- Сервер объединяет изменения за тик и ограничивает burst по budget на игрока.
- При нарушении контракта сервер отклоняет действие и принудительно высылает корректирующий sync.

### Rollout и rollback
- По умолчанию управление через feature flag (`inventory.ui.replacement.enabled`).
- Поэтапный rollout: dev -> staging -> prod.
- Kill switch возвращает открытие ванильного экрана без миграции данных и без потери предметов.

### Наблюдаемость
Минимум метрик по UI-слою:
- `inventory_ui_actions_total` (по типам действий),
- `inventory_ui_action_reject_total` (по причинам отказа),
- `inventory_sync_partial_total` / `inventory_sync_full_total`,
- `inventory_ui_rollback_total`.

## 7. API для других модов

Это одна из ключевых новых требований.

### Что нужно дать внешним системам
Серверный API, через который мод может:
- получить снимок доступных слотов игрока,
- найти предметы по фильтру,
- проверить наличие ресурса,
- зарезервировать/извлечь предметы транзакционно,
- вставить предметы в совместимые слоты,
- запросить только чтение или чтение+запись.

### Чего нельзя делать
- Нельзя разрешать внешним модам напрямую ковырять внутренние структуры GUI.
- Нельзя отдавать ссылку на внутренний `ItemStackHandler` без ограничений.
- Нельзя позволять изменения вне серверного потока.

### Production-решение
Нужен сервисный слой, например:
- `ExtendedInventoryService`
- `InventoryTransactionService`
- `EquipmentLayoutService`
- `AdminInventoryService`

Внешний мод вызывает не raw storage, а **контролируемые серверные операции**.

### Пример сценария
Мод на оружие хочет установить модуль:
1. Делает серверный запрос в API.
2. API ищет модуль в разрешённых слотах.
3. Если предмет найден и операция валидна — выполняется транзакция.
4. После применения сервер помечает инвентарь dirty и отправляет нужный sync.

## 8. Производительность и сетевая нагрузка

Это теперь один из главных архитектурных факторов.

### Главное правило
**Не отправлять инвентарь постоянно.**

### Нельзя
- синкать всё по тикам,
- отправлять пакет на каждый чих клиента,
- пересчитывать весь layout и весь инвентарь каждый тик для каждого игрока,
- делать полную синхронизацию после любой мелочи.

### Нужно
Использовать **event-driven sync**:
- полная синхронизация при входе,
- полная синхронизация при открытии меню,
- полная синхронизация при радикальной смене layout,
- дифф или пакет изменённых слотов при локальных изменениях,
- синхронизация только тем клиентам, которым это реально нужно.

### Рекомендуемая схема
- У каждого инвентаря есть `revision`.
- Все изменения помечают конкретные dirty slots/groups.
- В конце серверного тика или по событию отправляется **один агрегированный пакет**.
- Если за тик изменилось 5 слотов, отправляется один пакет с 5 слотами, а не 5 отдельных пакетов.

### Когда отправлять full sync
- login,
- respawn/clone,
- dimension change,
- open menu,
- изменение конфигурации layout,
- надет/снят предмет экипировки, который меняет количество или типы слотов.

### Когда отправлять partial sync
- изменение содержимого нескольких конкретных слотов,
- изменение одного предмета,
- операция через API другого мода,
- действие администратора.

### Дополнительные меры
- Не пересчитывать layout каждый тик — только при изменении экипировки/конфига.
- Кэшировать resolved layout игрока.
- Пересобирать layout только по dirty event.
- Не сериализовать всё NBT заново без необходимости.

## 9. Админский GUI и офлайн-редактирование

Это отдельный важный блок, который сильно влияет на архитектуру хранения.

### Функции админского GUI
- открыть инвентарь игрока по имени/UUID,
- просмотреть экипировку и контейнеры,
- изъять предмет,
- выдать предмет,
- переместить предмет между слотами,
- очистить запрещённый слот,
- видеть, онлайн игрок или офлайн.

### Если игрок онлайн
Работа идёт через серверный runtime-инвентарь и после операции сразу идёт sync.

### Если игрок офлайн
Работа идёт через `SavedData`, без загрузки сущности игрока в мир.

### Критически важно
Любое админское действие должно:
- проверять permission level,
- логироваться в audit log,
- быть транзакционным,
- обновлять `revision`,
- уметь корректно обработать конфликт, если игрок вошёл во время редактирования.

### Нужна стратегия конфликтов
Если админ редактирует офлайн-данные, а игрок в этот момент заходит:
- данные должны блокироваться на время транзакции,
- либо должна быть проверка версии и повтор операции,
- либо вход игрока должен перечитать уже обновлённое состояние.

Простейший production-подход:
- операции делать на главном серверном потоке,
- использовать version check,
- после записи офлайн-данных invalidation кэша,
- при входе всегда грузить последнюю сохранённую ревизию.

## 10. Логика экипировки и риски при смене layout

Самый опасный сценарий здесь — предмет экипировки снимается, а выданные им слоты были заняты.

### Например
Игрок снимает разгрузку, которая давала 8 слотов, а в этих слотах лежат магазины.

### Правильное поведение
Операция снятия должна быть **транзакционной**:
1. Сервер проверяет, можно ли переразместить все предметы.
2. Если можно — применяет перенос.
3. Если нельзя — снимает отказ и не завершает действие.

### Плохая идея
Автоматически выбрасывать предметы в мир по умолчанию.

### Хорошая идея
Иметь конфиг поведения:
- `deny_remove_if_overflow`
- `move_to_available_slots_first`
- `allow_overflow_readonly`
- `drop_to_world_last_resort` — только опционально

### Лучший безопасный дефолт
Если после снятия экипировки некуда перенести содержимое, **снятие запрещается**.

## 11. Основные технические риски

### 11.1. Packet flood
Риск:
- слишком частые пакеты от GUI,
- полные sync-пакеты вместо частичных,
- синки на каждый тик.

Снижение:
- event-driven sync,
- dirty batching,
- revision-based partial updates,
- запрет на full sync без причины.

### 11.2. Race conditions
Риск:
- GUI игрока,
- экипировка,
- сторонний мод,
- админский GUI
все пытаются менять один инвентарь почти одновременно.

Снижение:
- все мутации только на серверном потоке,
- транзакционный сервис,
- version/revision checks,
- единая точка записи через service layer.

### 11.3. Offline admin edits
Риск:
- редактирование данных офлайн-игрока конфликтует с его входом,
- повреждение данных при прямой правке NBT.

Снижение:
- хранить данные через `SavedData`,
- не править player `.dat` вручную,
- вести audit log,
- проверять revision на запись.

### 11.4. Миграции конфигов и слотов
Риск:
- изменился конфиг,
- переименовались slot types,
- изменился layout,
- предметы оказались в больше несуществующих слотах.

Снижение:
- хранить `schemaVersion`,
- хранить `layoutVersion`,
- иметь миграции,
- поддерживать overflow/read-only режим для устаревших слотов.

### 11.5. Потеря предметов при смене экипировки
Риск:
- удаление или дроп из-за неполной логики переноса.

Снижение:
- snapshot + dry-run + commit/rollback,
- безопасный отказ по умолчанию.

## 12. Что я бы вычеркнул из дальнейшего проектирования прямо сейчас

Чтобы не расползаться по лишним направлениям, пока не включать в ядро этого мода:
- систему генерации лута,
- механику редких материалов,
- зависимость дизайна инвентаря от чужих storage-модов,
- привязку архитектуры к фиксированному числу слотов,
- логику крафта как центральную часть первой версии.

Эти вещи можно интегрировать потом как отдельные потребители API, но не как фундамент inventory core.

## 14. Приоритетный roadmap

### Этап 1. Foundation
Сделать:
- capability игрока,
- `SavedData` по UUID,
- сериализацию NBT,
- lifecycle login/logout/clone,
- dirty-flag и revision.

Результат:
- данные живут на сервере,
- инвентарь сохраняется стабильно,
- возможна база для офлайн-редактирования.

### Этап 2. Универсальные слоты и layout
Сделать:
- `SlotDefinition`,
- `SlotGroupDefinition`,
- базовый layout resolver,
- поддержку динамических слотов от экипировки.

Результат:
- нет жёсткой привязки к 6 слотам,
- GUI масштабируется под разные комплекты экипировки.

### Этап 3. Транзакции и безопасность
Сделать:
- snapshot/dry-run/rollback,
- безопасное снятие экипировки,
- политики overflow,
- серверную валидацию всех действий.

Результат:
- защита от потери предметов и грязных состояний.

### Этап 4. Синхронизация без перегруза
Сделать:
- full sync на вход/открытие/пересборку layout,
- partial sync по dirty slots,
- batching изменений,
- запрет на бессмысленный spam sync.

Результат:
- нормальная работа на сервере без лишней сетевой нагрузки.

### Этап 5. API для других модов
Сделать:
- серверные методы чтения/изменения,
- фильтры поиска предметов,
- транзакционные операции,
- документацию по интеграции.

Результат:
- оружейные, медицинские и другие моды работают через безопасную точку входа.

### Этап 6. Админский GUI
Сделать:
- поиск игрока,
- онлайн/офлайн режим,
- просмотр и изъятие предметов,
- audit log,
- проверку прав.

Результат:
- администратор может решать дисциплинарные кейсы без ручного редактирования сейвов.

### Этап 7. Production UI Replacement
Сделать:
- заменить открытие ванильного инвентаря на `ExtendedInventoryScreen`,
- внедрить action-contract `C2SInventoryActionPacket` с полной серверной валидацией,
- включить transaction boundary на все пользовательские операции,

- ввести feature flags + kill switch + rollout/rollback runbook.

Результат:
- игрок видит новый production UI как основной экран инвентаря,
- сервер остаётся источником истины,
- поведение устойчиво под нагрузкой и операционно управляемо.

## 15. Ключевые решения, которые стоит принять сразу

1. **Не переписывать ванильный инвентарь на низком уровне**, а строить поверх него единый расширенный слой.
2. **Хранить persistent-данные не только в capability, но и в `SavedData`**, иначе офлайн-админка получится хрупкой.
3. **Сделать универсальную модель слотов**, а не фиксированную сетку.
4. **Вынести всю бизнес-логику в сервисы**, а не размазывать её по GUI и пакетам.
5. **Сделать событийную синхронизацию с dirty batching**, а не постоянную отправку пакетов.
6. **Сразу заложить revision/schema version/layout version**.
7. **Дать внешний API только через контролируемые серверные операции**.
8. **Делать замену пользовательского инвентарного UI через feature-flag rollout и rollback, а не как MVP-ветку без операционного контроля**.

## 16. Итог

После уточнений проект выглядит уже не как «ещё один контейнер на 6 слотов», а как **ядро серверной системы хранения и экипировки игрока**.

Значит, основу нужно строить не вокруг GUI, а вокруг:
- серверного хранения,
- динамического layout,
- транзакций,
- API для внешних систем,
- офлайн-администрирования,
- экономной синхронизации.

Именно такая архитектура даст:
- масштабирование под разные типы экипировки,
- безопасную работу с модами оружия/медициной/радиацией,
- нормальную нагрузку на сервер,
- возможность развивать систему дальше без переписывания базы.
